Проект находился на стадии базовой работоспособности, была настроена сущность пользователя, подключена БД через Liquibase и реализован вход через OAuth. Однако система авторизации по логину/паролю работала некорректно (тестовые пользователи из БД не могли войти), архитектура сервисов была немного спутана, а панель администратора отсутствовала. 

Что было сделано:

1. Глубокая отладка Spring Security:
   Проведен дебаг процесса аутентификации. Выявлена проблема несовпадения хэшей BCrypt из SQL-скрипта и текущей версии кодировщика в проекте.
   Написан временный `CommandLineRunner` для генерации валидного хэша прямо в контексте приложения, после чего хэши в БД были обновлены. Проблема `Bad Credentials` успешно решена.

2. Исправление авторизации ролей и статусов:
   Переписан метод `loadUserByUsername` в `CustomUserDetailsService`. Теперь Spring Security корректно подхватывает роли (`ROLE_ADMIN` / `ROLE_USER`) из базы данных.
   В конструктор `User` для Spring Security добавлена проверка поля `enabled` из нашей БД. Теперь заблокированные пользователи физически не могут пройти аутентификацию (выбрасывается исключение отключенного аккаунта).

3. Рефакторинг архитектуры (Чистый код):
   Устранены нарушения SOLID: методы по работе с пользователями (`getAllUsers`, `toggleUserStatus`) были ошибочно связаны с `FileService`.
   Проведен рефакторинг: логика строго разделена. Файловый сервис отвечает только за файлы, юзер-сервис — только за пользователей. Устранены все ошибки компиляции, связанные с внедрением зависимостей (`UserRepository`).

4. Реализация Панели Администратора:
   Создан `AdminController` и защищен маршрут `/admin/**` на уровне `SecurityConfig` (доступ только для `ROLE_ADMIN`).
   Разработан frontend-шаблон `admin.ftlh` с использованием Bootstrap для вывода таблицы всех пользователей.
   Внедрен функционал "Бан-молот": добавлена форма и метод в сервисе для инверсии статуса пользователя (`enabled` = `true/false`). Кнопка динамически меняет цвет и текст в зависимости от статуса.

5. Работа с файловой системой:
   Выявлен и задокументирован баг с путями к тестовым файлам при переносе проекта (Issue: абсолютные пути vs относительные после распаковки ZIP). 
   Тестовые данные сброшены, пути пересозданы, загрузка и скачивание новых файлов протестированы и работают корректно.

Потраченное время 5-6 часов
